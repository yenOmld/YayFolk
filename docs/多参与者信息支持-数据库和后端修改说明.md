# 多参与者信息支持 - 数据库和后端修改说明

## 概述
为了支持活动报名时填写多个人员信息，需要在数据库和后端进行相应的修改。

## 数据库修改

### 新建表：activity_reserve_participants
创建了一个新的表来存储每个订单的参与者信息。

**表结构：**
```sql
CREATE TABLE IF NOT EXISTS `activity_reserve_participants` (
  `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `reserve_id` BIGINT NOT NULL COMMENT '预订ID，关联activity_reserves表',
  `name` VARCHAR(50) NOT NULL COMMENT '参与者姓名',
  `phone` VARCHAR(20) NOT NULL COMMENT '参与者电话',
  `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  KEY `idx_reserve_id` (`reserve_id`),
  CONSTRAINT `fk_reserve_participants_reserve` FOREIGN KEY (`reserve_id`) REFERENCES `activity_reserves` (`id`) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='活动预订参与者信息表';
```

**字段说明：**
- `id`: 主键ID
- `reserve_id`: 关联到 activity_reserves 表的外键
- `name`: 参与者姓名
- `phone`: 参与者电话
- `create_time`: 记录创建时间
- `update_time`: 记录更新时间

**外键约束：**
- 当 activity_reserves 表中的记录被删除时，对应的参与者记录也会被级联删除

## 后端修改

### 1. 新建实体类：ActivityReserveParticipant
**文件路径：** `src/main/java/com/yayfolk/backend/entity/ActivityReserveParticipant.java`

**主要字段：**
- `id`: 主键
- `reserveId`: 预订ID
- `name`: 参与者姓名
- `phone`: 参与者电话
- `createTime`: 创建时间
- `updateTime`: 更新时间

### 2. 新建Repository接口：ActivityReserveParticipantRepository
**文件路径：** `src/main/java/com/yayfolk/backend/repository/ActivityReserveParticipantRepository.java`

**主要方法：**
- `findByReserveId(Long reserveId)`: 根据预订ID查找所有参与者
- `deleteByReserveId(Long reserveId)`: 根据预订ID删除所有参与者

### 3. 修改MerchantService类
**文件路径：** `src/main/java/com/yayfolk/backend/service/MerchantService.java`

#### 3.1 添加依赖注入
在构造函数中添加了 `ActivityReserveParticipantRepository` 的依赖注入。

#### 3.2 修改bookActivity方法
在创建订单后，调用 `saveParticipants` 方法保存参与者信息。

#### 3.3 新增saveParticipants方法
```java
private void saveParticipants(Long reserveId, Map<String, Object> data, User user)
```

**功能说明：**
- 接收前端传递的参与者数组
- 解析参与者信息并保存到数据库
- 兼容旧数据：如果没有传递参与者数组，则使用单个参与者的方式保存
- 错误处理：如果解析失败，回退到单个参与者的保存方式

**数据格式：**
```json
{
  "participants": [
    {
      "name": "张三",
      "phone": "13800138000"
    },
    {
      "name": "李四",
      "phone": "13900139000"
    }
  ]
}
```

#### 3.4 修改buildUserBookingDetailMap方法
在返回订单详情时，从数据库中查询并返回所有参与者信息。

#### 3.5 修改buildMerchantBookingDetailMap方法
在返回商家端订单详情时，同样返回所有参与者信息。

### 旧数据处理
现有订单的参与者信息存储在 `activity_reserves` 表的 `contact_name` 和 `contact_phone` 字段中。为了保持兼容性：

1. 新的 `saveParticipants` 方法会自动处理没有参与者数组的情况
2. 前端会检查 `participants` 数组，如果为空则使用 `contact_name` 和 `contact_phone` 显示单个参与者

## 注意事项

1. **数据一致性**：确保参与者数量与订单的 `participant_num` 字段一致
2. **外键约束**：删除订单时会自动删除对应的参与者记录
3. **兼容性**：保持了与旧数据的兼容性，不会影响现有功能
4. **错误处理**：添加了适当的错误处理，确保数据解析失败时能够回退到旧的处理方式
